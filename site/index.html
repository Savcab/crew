01 the idea

Agents are nodes. Relationships are edges.
You compose the team.

The classic setup is one hard-coded shape: a manager hands tasks to workers. crew throws that out. You get a blank graph and draw any team — then crew runs it.

+ Agent

Click + Agent on the dashboard (or crew spawn-agent). It gets a home directory, a tmux session, a launched Claude, and an identity.md.

Connect

Click two agents and describe the edge: what each does, and the condition under which the source should message the target — all in plain language.

Watch them work

Click any node to drop into its live terminal — a real tmux attach streamed to the browser. They coordinate by messaging along the edges you drew.

02 the gate

An agent can only message the agents it's connected to.

Not a UI suggestion — a hard wall enforced at delivery. crew message <peer> types into the peer's prompt only if an edge authorizes it. No edge, no message.

✓ along an edge

# inside the `leads` session
$ crew message builder \
   "Acme Plumbing, no site — build a demo"
[crew] sent to 'builder'

✕ no edge

# leads has no edge to sales
$ crew message sales "hey"
[crew] BLOCKED: 'leads' has no
relationship to 'sales' — connect
them first.

A relationship is directed by default (leads → builder lets leads message builder, not the reverse) — or mark it two-way. The condition you wrote ("when a qualified lead is found") is handed to the agent so it knows when to reach out.

03 durable identity

Claude sessions are ephemeral. Crew agents aren't.

An agent is a persistent identity = one home directory + one tmux session + an identity.md. If a session restarts, Claude re-reads the file and resumes who it is.

One agent per directory

crew refuses to put an agent inside another agent's home — or to share one. Homes never overlap, so two agents' work can't collide on disk.

identity.md

Written into every home: the agent's role, its workspace boundary, and the exact list of peers it may message — each with the condition you set. The durable answer to "who am I?".

# /Users/you/crew/leads/identity.md
# Identity: leads
**Role:** finds businesses with no website

## Who you may talk to
You may message ONLY these agents (delivery to anyone else is blocked):
- builder — builds demo sites
  - message them when: when a qualified lead with contact info is found
04 how it works

MorphDB for the graph. A tmux PTY bridge for the terminals.

Data → MorphDB

Two types: agent and edge (a first-class object with source/target relations, so it carries the description + condition + direction).

The whole gate is one index-backed query: GET /objects/edge?source=A&target=B. Schema morphs instantly as the model grows — no migrations.

Terminals → tmux PTY

Each agent is a real claude in a tmux session. The dashboard streams a true tmux attach client over a PTY to xterm.js — native scrollback, resize, mouse, the works.

Pure Python stdlib server. No build step. No runtime third-party deps.

 browser  ── xterm.js glass graph
    │  HTTP / SSE
 crew :8788  ── /api/graph/snapshot · /api/agent/* · /api/edge/* · /api/pty/*
    │
    ├── MorphDB :8787  ── agents + edges
    └── tmux           ── one live Claude session per agent
05 quickstart

From zero to a working crew.

# the data backend (https://morphdb.pages.dev)
$ morphdb start

# schema + dashboard → http://127.0.0.1:8788
$ crew init

# create agents — each gets a home + tmux session + claude
$ crew spawn-agent leads   --role "finds businesses with no website"
$ crew spawn-agent builder --role "builds demo sites"

# connect them — and say WHEN leads should message builder
$ crew connect leads builder \
    --when "when a qualified lead with contact info is found"

# open the glass dashboard, click a node, watch them go
$ crew dashboard open
